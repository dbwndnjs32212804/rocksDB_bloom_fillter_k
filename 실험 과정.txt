1.  rm -rf /tmp/rocksdbtest-1000/dbbench
데이터베이스 오염 (기존 데이터 간섭)을 물리적으로 완전히 차단하는 명령어

rm (Remove): 파일이나 디렉터리(폴더)를 삭제하는 리눅스 명령어
-r (Recursive): 디렉터리 안쪽에 있는 모든 하위 폴더와 파일(SSTable, WAL, 로그 등)까지 연쇄적으로 싹 다 지우는 옵션
-f (Force): 파일이 없거나 권한 문제가 있더라도 묻지 않고 강제로 삭제하라는 옵션
/tmp/rocksdbtest-1000/dbbench: db_bench가 테스트 데이터를 저장하는 기본 디렉터리 경로

-> 터미널(OS) 단에서 기존 DB 폴더 자체를 흔적도 없이 날려버리는 확실한 청소 작업. 
새로운 m/n이나 k 값을 테스트할 때 완벽한 백지상태(Clean State)를 만들어 준다.
 

2.  make -j4 db_bench
블룸 필터 소스 코드(bloom_impl.h)의 k 값을 수정한 뒤, 수정된 코드를 빠르게 적용하여 실행 파일을 재컴파일하는 명령어

make: 소스 코드를 컴파일하고 링크해서 실행 파일로 만들어 주는 기본 빌드 도구
-j4 (Jobs 4): 컴파일 작업을 한 줄로 처리하지 않고 4개의 작업(스레드)으로 쪼개서 동시에 병렬로 처리.
컴파일 속도가 비약적으로 빨라진다.
db_bench: 전체 RocksDB 코드를 전부 컴파일하면 시간이 너무 오래 걸리니, 지금 당장 실험에 필요한 db_bench 벤치마크 프로그램만 타겟으로 지정해서 빌드



3.  time ./db_bench --benchmarks="fillrandom,readrandom,readmissing" --num=10000000 --bloom_bits=30 --write_buffer_size=67108864 --max_background_jobs=4 --statistics --use_existing_db=false > result_10M_30_10.txt
벤치마크 실행 명령어

실행 옵션
time 
./db_bench 작업이 완료될 때까지 걸린 소요 시간을 측정

./db_bench     --benchmarks
--benchmarks="fillrandom,readrandom,readmissing"
이전과 동일하게 데이터를 넣고(fill), 읽고(read), 의도적으로 없는 데이터를 찾아(missing) 블룸 필터의 방어율을 테스트

--num=10000000 
총 1,000만 개의 Key-Value 데이터를 대상으로 벤치마크를 수행 

--bloom_bits=30 
데이터 1개당 할당하는 블룸 필터 비트 수(m/n)를 30으로 설정

--write_buffer_size=67108864 
67108864 (64MB): MemTable 버퍼 크기

--max_background_jobs=4
병목 현상을 막기 위해 백그라운드 파일 정리 작업 (컴팩션)에 스레드 4개를 할당

--statistics
FPR 계산을 위한 useful, full.positive, true.positive 등의 상세 지표를 로그에 남김

--use_existing_db=false 
이전 실험의 데이터베이스(SSTable 및 필터)를 전부 지우고 다시 시작하는 옵션

> result_10M_30_10.txt 
표준 출력 리다이렉션 기법.
db_bench가 터미널에 출력되는 통계 텍스트를 화면에 띄우지 않고, result_10M_30_10.txt라는 파일에 저장.
